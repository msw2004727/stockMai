# AGENTS.md

## 專案協作規則

1. 每一回合對話結束前，必須更新 `WORKLOG.md`。
2. `WORKLOG.md` 每筆需包含：
   - 日期時間（台灣時區，YYYY-MM-DD HH:mm）
   - 本回合完成事項（重點摘要）
   - 目前進度（Done / In Progress / Blocked）
   - 下一步（下一個最小可執行動作）
3. 若使用者中斷或誤關對話，下一回合必須先讀 `WORKLOG.md` 最新一筆，再繼續執行。
4. 工作日誌重點要白話、短句，避免只寫抽象描述。
5. 若本回合無程式變更，也要記錄「確認結果」與「建議下一步」。

## 模組化規則（維護優先）

1. 先分層再寫功能：
   - `routes`: 只處理 HTTP 參數/回應與錯誤碼，不寫商業邏輯。
   - `service`: 放流程編排與商業規則。
   - `provider/client`: 外部系統呼叫（API、DB、第三方服務）。
   - `parser/mapper`: 資料轉換、欄位正規化。
   - `constants/schemas`: 常數與資料結構定義。
2. 單一職責：一個檔案只做一件主題，避免「路由 + 邏輯 + 外部呼叫」混在同檔。
3. 檔案大小控制：
   - 單檔建議不超過 300 行。
   - 超過 300 行時，優先拆出 `service` 或 `composable/helper`。
4. 函式大小控制：
   - 單函式建議不超過 60 行。
   - 複雜分支改為小函式組合。
5. 命名規範：
   - 以可讀性為先，名稱要能看出層級角色（例如 `get_history`, `fetch_twse_month`, `parse_daily_row`）。
6. 前端規範：
   - `components` 負責呈現。
   - 狀態與計算邏輯優先抽到 `composables` 或 `api` 模組。
7. 禁止把臨時測試碼、一次性腳本、偵錯輸出長期留在主流程檔案。
8. 重構後必做自我驗收：
   - API 端點至少手動驗證成功/失敗案例各一。
   - 前端至少執行一次 build。
   - 確認既有路徑、參數與回傳欄位相容（除非明確公告 breaking change）。

## 作業模式規則（雲端優先）

1. 預設採用「雲端作業模式」執行專案（Render/GitHub 線上服務優先）。
2. 規劃與實作時，優先提供不依賴本機 Docker 的流程，方便跨設備接續工作。
3. 僅在必要時才要求本機開發模式，且需同時提供「雲端替代方案」。
4. 若需使用本機工具（Docker Desktop、本機 DB、本機 Redis），必須先說明原因與最小操作步驟。
5. 部署與驗收以雲端環境為主，確認項目至少包含：`/health`、資料庫連線、核心 API 可用。
